# Capability Implementation Status

Status: `full_vertical_closure_chain_defined_needs_owner_review`

This registry is an execution ledger, not a roadmap essay. It uses acceptance types first, then Core, UI Binding, Blackbox, Artifact, Event, Governance, Restart, Release, and evidence status across P0/P1/P2/Release. User-visible abilities prioritize blackbox acceptance; composite internal substrates must be linked to the user paths they support.

External project routing is governed by `docs/governance/EXTERNAL_PROJECT_REGISTRY.md` and the external runtime reference queue under `docs/governance`. That governance registry classifies projects as `real_integration`, `absorb`, `learn`, `reference`, or `reject`; this capability registry only records the resulting P0/P1/P2 gates and acceptance status.

## Acceptance Type Rule

- `user_blackbox`: user can operate it; blackbox is first priority and Core + Blackbox must pass before close is allowed.
- `core_only`: pure internal substrate with no direct user path and no cross-path product obligation; Core evidence is the acceptance basis.
- `artifact`: durable output such as report, package, export, or snapshot; Artifact lifecycle evidence is required.
- `governance`: registry, queue, classification, status vocabulary, or policy; consistency checks are required.
- `composite`: internal substrate carried by multiple user paths; it does not need a standalone UI blackbox, but it must have linked blackbox cases that prove the substrate is exercised by real product paths.

## Status Vocabulary

- `core_status`: `not_started`, `partial`, `passed`, or `blocked`.
- `blackbox_status`: `not_required`, `linked_required`, `linked_partial`, `linked_passed`, `not_started`, `passed`, or `blocked`.
- `artifact_status`: `not_required`, `not_started`, `partial`, `passed`, or `blocked`.
- `event_status`: `not_required`, `not_started`, `partial`, `passed`, or `blocked`.
- `governance_status`: `not_required`, `not_started`, `partial`, `passed`, or `blocked`.
- `ui_binding_status`: `not_required`, `not_started`, `partial`, `passed`, `blocked`, `linked_required`, `linked_partial`, or `linked_passed`.
- `restart_status`: `not_required`, `not_started`, `partial`, `passed`, `blocked`, `linked_required`, `linked_partial`, or `linked_passed`.
- `release_status`: `not_started`, `blocked`, `p0_release_gate_passed_needs_owner_review`, `p1_release_gate_passed_needs_owner_review`, `p2_release_gate_passed_needs_owner_review`, or `overall_industrial_landing_candidate_needs_owner_review`.
- `release_blocker`: always `true` until the relevant staged Release Gate and Owner Review.
- `close_allowed`: whether this row has enough evidence for its own acceptance type before Owner Review; it is not a release claim.
- `evidence_report` and `evidence_commit`: report path and commit evidence for the row; `none` means the row is still queued or unverified.

## Routing Summary

- External project classification policy lives in `docs/governance/EXTERNAL_PROJECT_REGISTRY.md`; classification summary lives in the external runtime reference queue under `docs/governance`; classification does not replace capability acceptance.
- Stage chain: P0 backfill -> P0 Core Acceptance -> P0 Release Gate -> P1 -> P1 Release Gate -> P2 -> P2 Release Gate -> Final Owner Review.
- P1 queue must keep memory/evidence, task routing, Office adapter, AI governance, Loop and harness gates, but none should run before P0 dual-track acceptance.
- P2 queue must keep workgroup/A2A, night maintenance, company brain and release-adjacent industrial gates, but none should run during P0.
- `global_goal_complete` must remain `false` while any P0/P1/P2/Release gate remains.
