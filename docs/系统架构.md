# 系统架构

当前系统架构以 [产品架构 v3](product/PRODUCT_ARCHITECTURE_V3_2026-06-19.md) 为准。本文是 Core 仓库的落地说明。

## 总体层次

```text
L0 工作区层 Workspace
L1 资料接入层 Import Source
L2 文档库层 Document Library
L3 标准知识包 / OKF 候选层
L4 知识库层 Knowledge Base
L5 索引层 Index Layer
L6 RAG 与验证层 Retrieval / RAG / Validation
L7 编排层 Orchestration
L8 产物层 Artifacts
L9 治理配置层 Governance / Settings
```

主链路必须统一写为：

```text
文档库 → 知识库 → 索引层 → RAG → 编排层 → 文档/Skill/Agent/A2A
```

## Core 仓库落地

- Core CLI：`python -m heitang_kb_forge.cli`
- 数据合同：JSON、JSONL、Markdown、Office 文档输出
- 文档库：保存解析、清洗、结构化后的内容资产
- 知识库：由文档库或标准知识包候选层构建，保留 manifest、chunks、quality report、版本和来源
- 索引层：关键词索引、向量索引、元数据索引、引用索引、记忆索引
- RAG 层：查询改写、召回、重排、引用、验证、人工纠偏和验证报告
- 编排层：按目标调度 KB、索引、模板、模型、Provider、Skill、Agent、A2A
- Provider 层：LLM、Embedding、Parser、OCR、Redis、向量库、Search、Exporter 均按配置启用

## 工程兼容

公开产品名称可以调整为 HeiTang Knowledge Workbench，但 Python 包路径保持 `heitang_kb_forge`。Workbench UI 是表现层；Core/UI/Bridge 的历史 Campaign 名称只作工程追溯，不作为当前产品架构事实。

## 禁止误写

- 不把 UI handoff 写成 Campaign 4 UI 完成。
- 不把 Bridge handoff 写成 Campaign 5 Bridge 完成。
- 不把 Agent package 写成 executable runtime。
- 不把 Redis / Vector DB future target 写成当前 runtime。
- 不新增 OKF runtime、OKF 一级页面或 OKF 用户主链路。
