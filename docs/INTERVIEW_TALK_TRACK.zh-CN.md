# 面试讲法

## 30 秒版本

我做了一个面向 Agent 构建的知识供应链 Skill，叫 HeiTang KB Forge。它可以把 PDF、DOCX、Markdown、表格、图片等多格式资料，加工成标准化、可追溯、可检索、可评估、可导出的知识资产包，作为后续 RAG Agent、导购 Agent、教育 Agent、产品经理 Agent 的前置知识底座。

## 1 分钟版本

这个项目解决的是 Agent 构建里的上游知识治理问题。很多 Agent 效果不好，不一定是模型不行，而是进入模型前的知识没有标准化、没有证据追踪、没有质量门禁，也不方便复用。

所以我做了 HeiTang KB Forge。它支持多格式资料接入、知识包生成、质量门禁、证据治理、LLM Provider 安全检查、Agent / Skill 导出，以及一个可复现的 demo-e2e 闭环。

到 v2.7，它已经可以本地离线跑通一条演示链路：生成知识包、做质量检查、做 Provider 安全检查、用 mock LLM 做辅助质检、导出 Agent / Skill 包，并生成作品集报告和证据包。

## 3 分钟版本

HeiTang KB Forge 的定位不是普通 RAG Demo，也不是单纯文档解析工具，而是 Agent 构建前的知识供应链底座。

我观察到一个问题：企业或者个人有大量资料，但这些资料进入 RAG 或 Agent 系统前，往往缺少标准化、可追溯、可评估和可复用的加工流程。这会导致检索质量不稳定、回答不可控、后续维护成本高。

这个项目的解法是做一个本地优先的 CLI Skill，把多格式资料统一加工成标准知识资产包，并围绕它建立质量门禁、证据映射、Provider 治理、发布检查和导出能力。

我特别强调边界：默认离线和 mock，不默认联网；真实 LLM 是可选能力；API key 只从环境变量读取，不落盘；不把 mock 冒充真实平台发布。

v2.7 增加了最小端到端 Demo，能一键生成作品集报告和 evidence pack，让项目可以在 GitHub 和面试里直接展示。

## 高频问题

### 为什么做这个？

因为 Agent 的效果很大程度取决于上游知识质量。我想补的是 raw documents 到 Agent 之间缺失的知识资产生产层。

### 和普通 RAG 项目有什么区别？

普通 RAG 更关注检索和回答。这个项目更靠前，关注资料进入 RAG 前的标准化、质量、证据、治理、安全和导出。

### 产品价值是什么？

降低 Agent 构建的前置成本，让知识资产可复用、可追踪、可评估。

### 技术边界是什么？

它是本地 CLI / Skill 项目，不默认声明 SaaS、真实平台发布或生产级 Runtime 兼容。

### 下一步怎么做？

优先把最小 Demo 包装成更清晰的产品展示，再选择性补真实 Runtime 兼容和领域 Skill 模板。
