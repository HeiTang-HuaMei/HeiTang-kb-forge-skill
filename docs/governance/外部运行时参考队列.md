# 外部运行时参考队列

本队列只记录登记项目的分流结果，不代表外部项目已经进入当前
runtime。普通用户界面不得暴露外部项目、热插拔项目、插件市场、
Gate/Campaign 或 Core 操作概念。

## 当前规则

- 先判断登记项目是否值得吸收，再决定进入 Provider、模板资产或架构参考分流。
- 值得吸收的架构参考必须并行交付 contract、schema、runtime 边界、UI 信息架构边界、测试 gate、审计模型、降级策略或 Provider 分类/加载规则。
- 不值得吸收的登记项目必须写明拒绝原因，不得长期挂在参考队列。
- 无法立即吸收的登记项目只能使用 `deferred_with_blocker`，且必须有明确 blocker。
- Provider、Agent runtime、Computer Use、shell 或外部服务能力必须单独走安全和验收 gate。
- OKF 不在本队列实现 runtime；Stage3 只处理 Provider 化配置、加载边界、审计和回滚。

## 分流状态

| 状态 | 含义 |
| --- | --- |
| `capability_provider` | 能增强 Parser/OCR、Embedding/Vector、Search/Retrieval、Exporter、Agent 记忆/工具、A2A 导出或治理评测能力，且可配置、测试、审计、回滚。 |
| `template_asset` | 提供提示词、方法论、模板、写作框架或 Skill/Agent 范式，不加载外部 runtime。 |
| `absorbed_into_architecture` | 参考已转化为架构合同、schema、边界、gate、审计、降级或加载规则。 |
| `rejected_no_architecture_gain` | 对 v3 主链路无明确增益、成本高于收益、与架构冲突或已被现有能力覆盖。 |
| `deferred_with_blocker` | 有吸收价值，但存在明确 blocker；不得无限期保留。 |

## 外部项目分类

外部项目不能统一叫“能力来源”。每个项目必须先归入以下五类：

| 分类 | 含义 |
| --- | --- |
| `real_integration` | 真实接入为 Adapter、Connector、Runtime 依赖或内置工具，必须通过依赖、许可、本地部署、降级和黑盒验收。 |
| `absorb` | 不接代码、不接依赖，只把设计模式、数据结构、流程能力变成黑糖自己的实现。 |
| `learn` | 学习工程方法、组织方式或评测方式，不生成产品模块。 |
| `reference` | 仅作为产品形态、UI、市场或概念对照，不进入实现队列。 |
| `reject` | 明确不采用，并记录原因，避免反复讨论。 |

## 吸收验收规则

`absorb`、`learn` 和 `reference` 不能只靠登记表闭口。登记只说明外部项目不会直接变成黑糖产品模块；真正吸收必须在对应 P2 能力闭口时证明黑糖能力变强。

最小吸收证据字段：

| 字段 | 含义 |
| --- | --- |
| `extracted_strength` | 从外部项目提炼出的 1-3 个最强长处。 |
| `mapped_heitang_capability` | 这些长处归并到的黑糖既有能力。 |
| `native_design_change` | 黑糖原生设计、schema、流程、测试、审计或 UI 边界变化。 |
| `user_or_quality_improvement` | 用户体验、可靠性、治理或可维护性上的可观察改进。 |
| `evidence_report` | source_trace、validation_report、黑盒结果、回归报告或审计报告。 |
| `absorbed_status` | `absorb_candidate` / `absorbed_into_design` / `absorbed_into_runtime` / `absorbed_with_evidence` / `rejected_after_review`。 |

顺序规则：

- 该要求延后到 P2 对应能力自然闭口时处理，不插队到当前 Gate。
- 不新增主 Gate，不改变 P2 能力链到 `P2 Release Gate` 的顺序。
- 不把外部项目名展示给普通用户；用户只看到黑糖自己的能力结果和下一步操作。
- 不接代码、不新增依赖、不接 runtime 的项目，不能写成 `absorbed_into_runtime`。
- 只有已有 evidence_report 并通过对应能力验收后，才能写 `absorbed_with_evidence`。

分类关系：

- 每个项目必须有一个主处理方式：`real_integration` / `absorb` / `learn` / `reference` / `reject`。
- 主处理方式决定是否允许真实接入、是否允许新增依赖、是否进入 runtime 验收。
- 次级标签可以并存，用于记录额外借鉴点，例如 `shape_reference`、`engineering_method`、`template_schema_reference`、`advanced_candidate`、`connector_candidate`。
- 例如，一个项目可以主分类为 `absorb`，同时带 `shape_reference`；但不能因此把项目名放到普通 UI，也不能跳过对应 P2 能力验收。
- 一个项目如果主分类为 `learn`，只能产出工程方法、测试矩阵、审计规则或执行规范；不能被写成用户功能完成。

产品进步验收：

| 维度 | 通过标准 | 不足以通过 |
| --- | --- | --- |
| 用户价值 | 用户能更清楚、更少步骤、更安全或得到更好结果。 | 只新增项目名、登记项或隐藏配置。 |
| 质量 / 可靠性 | 有测试、benchmark、validation_report、source_trace 或回归结果证明更正确、更可追溯、更稳定。 | 只有学习笔记或设计口号。 |
| 黑糖原生适配 | 落到既有黑糖能力、schema、workflow、template、connector 或治理规则。 | 新增外部项目形状的产品模块。 |
| 边界安全 | 不新增未批准依赖、不打包外部服务、不暴露密钥、不做本地模型训练或 GPU 需求。 | 只有未来接入声明，没有依赖、fallback 和验收边界。 |
| 证据完整 | 对应 P2 能力报告记录提炼长处、归并能力、原生变化、前后改进和证据路径。 | 只写 `absorb`、`learn` 或 `reference`。 |

## 分类规则

- 只有 `real_integration` 可以进入真实接入计划。
- `absorb` 只能提炼能力，不接代码。
- `learn` 只学习工程方法，不生成产品模块。
- `reference` 只作为对照，不进入实现队列。
- `reject` 必须明确拒绝原因，避免反复讨论。
- 不得为每个外部项目新增独立产品模块，必须归并到 OKF、HeiTang Memory & Evidence Layer、Knowledge Reliability Engine、Document / Skill / Artifact Generation、Agent Execution Harness、Capability Chain Runner、Workgroup / A2A 或 Industrial Release。

## 当前 P0 影响

- `P0-4B OKF Minimal Core Gate` 是黑糖内部 OKF candidate baseline，不是 Obsidian 或外部 OKF 接入。
- `P0-4C Agent Memory Minimal Core Gate` 只吸收 TencentDB Agent Memory 的任务记忆思想，不接 TencentDB，不新增 Node 22 依赖。
- `P0-5B Knowledge Reliability Minimal Core Gate` 仍在 Agent Memory Minimal Core 后继续执行。
- P1/P2 外部项目分类只入队，不得在 P0 提前实现。

## 当前结论

登记项目不能用学习笔记替代交付。Stage3 的有效交付必须让产品能力、
架构边界、测试 gate 或审计回滚模型变得更清晰；否则必须明确拒绝。
