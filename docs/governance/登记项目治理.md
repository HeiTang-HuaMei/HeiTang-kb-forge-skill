# 登记项目治理

登记项目必须先分流，再决定是否进入当前产品能力。不能把项目长期
写成学习笔记或模糊参考，也不能把未验证能力写成已接入。

## 分流状态

- `capability_provider`：能力增强型 Provider。必须具备配置、健康检查、
  runtime 状态、失败降级、审计和回滚边界。
- `template_asset`：模板或方法论资产。必须具备来源、版本、manifest、
  验证结果和 Skill/Agent/文档模板绑定边界，不加载外部 runtime。
- `absorbed_into_architecture`：值得吸收的架构参考，已并行落到
  contract、schema、runtime 边界、UI 信息架构、测试 gate、审计模型、
  降级策略或 Provider 分类/加载规则。
- `rejected_no_architecture_gain`：不提升 v3 主链路，成本高于收益，
  与架构冲突，或已被现有能力覆盖。
- `deferred_with_blocker`：有吸收价值但被明确 blocker 阻断。必须写明
  blocker 和解除条件，不允许无限期保留。

## 吸收判断

登记项目只有满足至少一项才允许进入吸收流程：

- 补当前架构短板。
- 降低复杂度。
- 增强扩展性。
- 提升稳定性、可审计或可回滚。
- 改善 Provider、RAG、Agent、A2A、配置或权限等核心抽象。

## 禁止写法

- 已接入。
- runtime ready。
- 当前默认启用。
- 官方集成完成。
- 自动进入 Provider / 插件市场 / 工具链。
- 仅以 `reference_only`、`needs_verification` 或学习笔记作为长期状态。

## 使用方式

Provider 配件进入设置和下游能力状态；模板资产进入 Skill/Agent/文档
模板资产库；架构参考必须吸收、拒绝或带 blocker 延期。普通用户只
感知增强后的自然入口，不感知外部项目或热插拔工程概念。
