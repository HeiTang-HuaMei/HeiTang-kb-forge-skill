# 迁移与兼容策略

## 目的

本文件定义 schema、workspace、知识库和成果在版本变化中的兼容策略。

## 基本原则

- 旧数据可读优先。
- 迁移必须可解释。
- 迁移失败不得损坏原数据。
- 不把旧工程证据误显示为普通成果。
- 每次 schema 变化必须有版本号。

## Schema Version

每个持久化文件应有：

```json
{
  "schema_version": "name.v1"
}
```

读取时必须处理：

- 当前版本。
- 旧版本。
- 缺失版本。
- 损坏文件。

## Workspace 兼容

打开旧 workspace 时：

- 能识别则正常显示。
- 部分识别则显示可恢复状态。
- 无法识别则提示导出诊断或新建工作区。

不得静默清空旧 workspace。

## KnowledgeBase 兼容

旧 KB 必须检查：

- manifest。
- chunks。
- source_map。
- source_docs。
- is_deleted。

缺少 source_map 时：

- UI 应提示来源追溯不完整。
- 不得伪装为完整可追溯。

## Artifact 兼容

旧成果读取时必须分类：

- 普通成果。
- 操作记录。
- 工程证据。
- 无法识别。

无法识别的文件不得进入普通成果页。

## 迁移流程

推荐：

1. 读取旧数据。
2. 写迁移预览。
3. 备份或保留原文件。
4. 写新文件。
5. 校验后台真值。
6. 写 migration record。
7. UI 显示结果。

## 回滚

迁移失败时：

- 保留原数据。
- 标记 failed。
- 显示用户可理解错误。
- 提供导出诊断。

## 验收

必须验证：

- 旧 workspace 可打开。
- 旧 KB 不误删。
- 旧成果不污染普通成果页。
- 迁移失败可解释。
- 重启后状态一致。
