# 开发工作流

## 目的

本文件定义从问题发现到修复完成的协作流程。

## 默认流程

```text
发现问题
-> 记录 bug
-> 确认 latest running UI
-> 定位代码和数据
-> 最小修复
-> 窄验证
-> running UI 复验
-> 后台真值对账
-> 重启恢复
-> 修复报告
```

## 分支和提交

当前侧重本地修复，不默认 push、tag、release。

提交前必须确认：

- 不混入无关 dirty 文件。
- 不改状态机，除非任务明确要求。
- 不提交密钥。
- 不提交 Owner 私有数据。
- 不把测试样本混入普通成果。

## Dirty Worktree

遇到已有 dirty 文件：

- 先识别是否与当前任务相关。
- 无关则不碰。
- 相关则读懂后在其基础上修。
- 不使用 destructive git 命令。

## 修复粒度

每次修复只覆盖一个清晰问题或一个连续任务链。

不建议：

- 一个提交同时修 UI、runtime、文案、重构、测试矩阵。
- 为了小文案改动重构大文件。
- 为了通过测试改状态机。

## 验证粒度

优先窄验证：

- 相关 widget test。
- 相关 runtime test。
- 相关 verifier script。
- running UI 对应路径。

最后按风险决定是否扩大回归。

## 汇报格式

```text
目标：
改动：
验证：
结果：
风险：
下一步：
```

## 停止条件

必须停止并重新确认：

- 可能删除真实用户数据。
- 需要改 release gate 或状态机。
- running UI 不是最新。
- 修复范围开始失控。
- 出现安全或密钥风险。
