# 黑糖 Knowledge Workbench PRD v3

日期：2026-06-19
定位：产品需求文档重构版
目标：在原 PRD v2 基础上，补齐索引层、RAG 层、编排层和登记项目治理，形成更完整的工业级产品需求。

---

## 1. 产品定义

黑糖 Knowledge Workbench 是一个本地优先的知识资产工作台，帮助用户把导入资料解析为文档库，再构建知识库、索引和 RAG 能力，最后通过编排层生成文档、Skill、Agent 和多 Agent 协作结果。

---

## 2. 产品目标

### 2.1 用户目标

用户不需要从零写提示词，也不需要反复翻资料。用户只需要导入资料，系统应完成：

```text
资料解析
文档库管理
知识库构建
索引构建
检索验证
文档生成
Skill 生成
Agent 创建
A2A 协作
产物导出
审计复验
```

### 2.2 产品目标

- 让资料变成可管理的文档资产。
- 让文档变成可组合的知识资产。
- 让知识库变成可检索、可验证、可调用的索引资产。
- 让 RAG 输出有来源、有引用、有验证、有纠偏。
- 让文档、Skill、Agent 都走同一条证据链。
- 让登记项目有参考、验证、接入边界，不混入当前 runtime。

---

## 3. 用户角色

| 角色 | 目标 | 重点能力 |
|---|---|---|
| 普通知识工作者 | 导入资料，生成可用文档 | 文档库、知识库、模板生成 |
| 内容创作者 | 基于资料生成文章、报告、脚本 | 多 KB、RAG、模板、导出 |
| 知识工程用户 | 构建可复用知识库和 Skill | 索引、验证、Skill 工厂 |
| Agent 搭建者 | 创建可对话、可协作 Agent | Agent 工作区、KB/Skill 绑定、A2A |
| Owner / 验收者 | 判断是否真实可用 | 产物、审计、重启一致性 |

## 4. 完整用户流程

### 4.1 首次启动与配置

用户目标：完成最小可用配置。

操作步骤：

1. 启动 EXE。
2. 创建默认工作区。
3. 选择本地存储路径。
4. 配置 LLM Provider。
5. 可选配置 Embedding Provider。
6. 可选配置 Redis。
7. 可选配置向量库。
8. 可选配置 Search Provider。
9. 可选配置导出器。
10. 保存配置并测试连接。

输出：

- `workspace_manifest`
- `provider_config`
- `storage_config`
- `redis_config`
- `vector_config`
- `exporter_config`
- `settings_test_report`

---

### 4.2 导入与解析流程

用户目标：把原始资料变成文档库内容。

支持导入来源：

- 单文件。
- 文件夹。
- PDF。
- DOCX。
- Markdown。
- TXT。
- 图片。
- 表格。
- HTML / 网页链接。
- 外部 Skill 文件夹。

操作流程：

```text
选择导入来源
  → 扫描文件
  → 生成导入任务
  → 类型识别
  → 调用 Parser Provider
  → 可选 OCR
  → 生成正文
  → 生成结构统计
  → 生成 chunks
  → 写入文档库
  → 写入解析报告
```

关键需求：

- 同名文件不能覆盖。
- 每个文档必须有唯一 `document_id`。
- 失败文件不能影响其他文件。
- 失败原因必须可见。
- 导入历史可删除，删除后刷新 / 重启不恢复。
- 解析状态必须真实，不允许假成功。

---

### 4.3 文档库流程

用户目标：管理已经解析处理后的文档内容。

文档库定位：

文档库不是原始文件夹，而是由导入文件经过解析、清洗、结构化后形成的内容资产库。

文档库能力：

- 多文档持久化。
- 分类筛选。
- 标题 / 来源 / 内容搜索。
- 文档预览。
- 正文查看。
- chunks 查看。
- 结构统计。
- 关联知识库查看。
- 多选文档创建知识库。
- 加入已有知识库。
- 删除文档。
- 删除影响提示。
- 重新解析。
- 重启一致性。

输出：

- `document_manifest`
- `parsed_body`
- `document_chunks`
- `parse_report`
- `document_to_kb_links`

---

### 4.4 标准化知识包流程

用户目标：让文档库内容具备跨平台、可版本化、可追溯的标准封装。

定位：

标准化知识包位于文档库和知识库之间。OKF 属于这一层的候选标准。

```text
文档库内容
  → 标准化 manifest
  → metadata
  → content files
  → source references
  → version
  → optional OKF package
```

需求：

- 当前版本不把 OKF 做成一级页面。
- 当前版本不默认实现 OKF runtime。
- 后续可支持 OKF Bundle 导出。
- 后续可支持 OKF → 知识库构建。

---

### 4.5 多知识库流程

用户目标：从文档库组合出多个用途不同的知识库。

示例：

```text
Document A → KnowledgeBase K1
Document B → KnowledgeBase K2
Document A+B → KnowledgeBase K3
```

创建流程：

1. 从文档库选择一个或多个文档。
2. 填写知识库名称。
3. 选择知识库类型。
4. 选择构建配置。
5. 选择索引配置。
6. 构建知识库。
7. 生成 manifest、chunks、quality report、index metadata。

知识库类型：

- 普通。
- 写作。
- 产品。
- 教育。
- 法规。
- Agent 专属。
- 自定义。

知识库操作：

- 创建。
- 复制。
- 合并。
- 拆分。
- 增量更新。
- 全量重建。
- 版本比较。
- 回滚。
- 删除。

---

### 4.6 索引层流程

用户目标：让知识库可检索、可追溯、可被 Agent 调用。

索引类型：

- 关键词索引。
- 向量索引。
- 元数据索引。
- 引用索引。
- 记忆索引。

操作流程：

```text
选择知识库
  → 选择索引配置
  → 选择 Embedding Provider
  → 可选外部向量库
  → 构建索引
  → 生成 index metadata
  → 写入索引构建报告
```

需求：

- 本地索引默认可用。
- 外部向量库按配置启用。
- 外部向量库失败时必须提示或回退。
- 删除 KB 后，索引不可再被下游选择。
- 重建 KB 后，索引版本同步更新。

---

### 4.7 检索、RAG 与验证流程

用户目标：验证知识库是否可用，并为文档 / Agent 提供证据。

流程：

```text
输入查询
  → 选择单 KB / 多 KB
  → 查询改写
  → 调用索引层
  → 召回 chunks
  → 重排
  → 展示引用证据
  → 可选外部事实验证
  → 保存人工纠偏
  → 生成验证报告
```

指标：

- 命中 chunk 数。
- 来源 KB。
- 来源文档。
- 引用覆盖率。
- 答案覆盖率。
- 重排分数。
- 冲突点。
- 外部验证结果。
- 纠偏记录。

要求：

- 检索前必须选择 KB。
- 多 KB 检索必须标明来源 KB。
- 命中 chunk 可查看。
- 引用可追溯到文档和 chunk。
- 外部事实验证受网络授权和 Provider 配置控制。
- 人工纠偏可保存、可审计。

---

### 4.8 文档生成流程

用户目标：基于知识库和模板生成可编辑、可导出的文档。

支持文档类型：

- PRD。
- 分析报告。
- 文章。
- 教案。
- 手册。
- 会议纪要。
- 自定义模板。

操作流程：

```text
选择输出类型
  → 选择模板
  → 选择单 KB / 多 KB
  → 编排层生成 RAG 上下文
  → 可选先生成大纲
  → 生成 Markdown 正文
  → 展示引用列表
  → 用户编辑
  → 保存历史快照
  → 导出
```

导出格式：

- Markdown 默认可用。
- DOCX 配置导出器后可用。
- PDF 配置导出器后可用。
- PPTX 配置导出器后可用。
- JSON / CSV 配置导出器后可用。

历史要求：

- 生成历史可查看。
- 单条历史可删除。
- 历史可重新打开。
- 历史快照不能被后续生成覆盖。
- 删除后刷新 / 重启不恢复。

---

### 4.9 Skill 工厂流程

#### 4.9.1 从知识库生成 Skill

用户目标：把知识库沉淀为可复用的方法论和工具规则。

流程：

```text
选择 KB
  → 选择 Skill 类型
  → 选择目标平台
  → 提取方法论
  → 提取输入输出
  → 提取规则边界
  → 生成 Skill 草稿
  → 用户编辑
  → 样例任务验证
  → 导出 / 绑定 Agent
```

Skill 类型：

- 问答型。
- 写作型。
- 分析型。
- 检索型。
- 教学型。
- 产品型。
- 自定义。

目标平台：

- Codex。
- Claude Code。
- OpenClaw。
- Markdown。
- Internal Agent。

#### 4.9.2 外部 Skill 本地化 / 个性化

用户目标：把外部 Skill 与本地知识库融合。

流程：

```text
导入外部 Skill S0
  → 解析 Skill 结构
  → 展示方法论
  → 选择本地 KB
  → 编排层融合 S0 + KB
  → 生成本地化 Skill S1
  → 生成个性化 Skill S2
  → 验证
  → 导出 / 绑定 Agent
```

要求：

- 外部 Skill 不能直接视为已适配。
- 必须记录来源、版本、平台、融合 KB。
- 必须生成治理报告。

---

### 4.10 Agent 工作区流程

#### 4.10.1 Agent 定义

Agent 是绑定了工作区、知识库、Skill、模型、记忆、工具和权限边界的可控智能体。

#### 4.10.2 简单模式

目标：快速创建可对话 Agent。

字段：

- Agent 名称。
- Agent 类型。
- 绑定 KB。
- 绑定 Skill。
- 模型。

要求：

- 创建后立即可对话。
- 回复显示 KB / Skill 来源。
- 保存对话历史。
- 可导出 Markdown。
- 重启后历史仍在。

#### 4.10.3 复杂模式

字段：

- Agent 名称。
- Agent 类型。
- 模型配置。
- 多 KB。
- 多 Skill。
- Tool。
- Redis 短期记忆。
- 向量库长期记忆。
- 权限策略。
- 审计策略。

要求：

- Tool 仅复杂模式展示。
- Redis / 向量库按配置启用。
- Agent 不得访问未授权 KB / Skill。

---

### 4.11 A2A 多 Agent 流程

用户目标：让多个 Agent 在总工作区内协作。

流程：

```text
创建多 Agent 总工作区
  → 创建子 Agent 工作区
  → 每个子 Agent 绑定自己的 KB / Skill
  → 输入协作议题
  → 总工作区分发任务
  → 子 Agent 输出
  → Agent 互评 / 回应
  → 汇总共识
  → 标记冲突点
  → 生成行动建议
  → 导出 A2A 报告
```

要求：

- A2A 必须在总工作区发生。
- 每个子 Agent 必须有独立工作区。
- 子 Agent 基于自身 KB / Skill 输出。
- 总工作区负责汇总，不直接篡改子 Agent 配置。
- A2A 不是单 Agent 的替代品。

登记项目不得直接进入产品主链路。状态必须按以下规则流转：

```text
reference_only
  → needs_verification
  → verified_candidate
  → integration_planned
  → integrated
```

## 8. 非功能需求

### 8.1 稳定性

- EXE 不白屏、不黑屏。
- 单文件失败不影响批量任务。
- Provider 超时有明确失败状态。
- Redis / 向量库连接失败不影响本地基础链路。

### 8.2 可恢复性

- 任务进度落盘。
- 最近任务真实可删除。
- 生成历史真实可恢复。
- 重启后工作区、文档库、知识库、配置、产物一致。

### 8.3 可观测性

- 导入审计。
- 解析审计。
- KB 构建审计。
- 索引构建审计。
- RAG 检索审计。
- 文档生成审计。
- Skill 生成审计。
- Agent 运行审计。
- A2A 协作审计。

### 8.4 安全性

- API Key 不明文展示。
- Redis 密码不明文展示。
- 向量库密钥不明文展示。
- Agent 不越权访问未绑定 KB / Skill。
- 外部 Provider 必须受网络授权控制。
- 不执行任意 shell。

### 8.5 性能

- 大文件导入有进度。
- 多文件导入可分批。
- 检索有超时和降级。
- 向量库不可用时有本地索引兜底。

## 10. 真实 EXE Smoke 流程

```text
启动 EXE
  → 创建普通工作区 W0
  → 配置 Provider
  → 可选配置 Redis / 向量库
  → 导入 A 文件
  → 导入 B 文件
  → 文档库显示 A+B
  → 选择 A 创建知识库 K1
  → 选择 B 创建知识库 K2
  → 选择 A+B 创建知识库 K3
  → K1/K2/K3 生成 index metadata
  → 检索 K1
  → 检索 K1+K2+K3
  → 用 K1 生成文档 D1
  → 重开 D1 历史快照
  → 用 K1 生成 Skill S1
  → 导入外部 Skill S0
  → 用 S0 + K2 生成本地化 Skill S2
  → 创建单 Agent 工作区 W_A
  → 创建 Agent A，绑定 K1 + S1
  → Agent A 对话成功并显示引用
  → 创建多 Agent 总工作区 W_M
  → 创建子 Agent B 工作区，绑定 K2 + S2
  → 创建子 Agent C 工作区，绑定 K3
  → 在 W_M 启动 A2A
  → 输出协作摘要
  → 重启 EXE
  → 验证数据仍一致
```

验收时如果这条链路断开，不能进入 stable。

## 11. 开发实现原则

1. 不做假功能。
2. 不用黄色 pending 掩盖主链路缺口。
3. 不把 reference queue 写成 integrated。
4. 不默认打包 Redis / 向量库服务本体。
5. 不默认引入 GPU / 视频 / 本地大模型。
6. 每个按钮必须有真实结果或明确失败原因。
7. 每个产物必须有路径、内容、来源、审计。
8. 每个生成链路必须支持重启一致性。
9. 能力补全优先于装饰性 UI。
10. 页面结构必须按用户链路排列。
